> **原文**: [A Practical Guide to Building Agents](https://cdn.openai.com/business-guides-and-resources/a-practical-guide-to-building-agents.pdf) | OpenAI, 2025年4月
>
> OpenAI 出的 Agent 构建指南，下面是我的逐章解读

---

## 一、Agent 到底是什么

### 1.1 一句话解释

**Agent 就是能替你干活的 AI 系统。** 你给它一个目标，它自己想办法完成，中间该查数据查数据，该调接口调接口。

它和普通的 ChatGPT 对话有两个本质区别：

- **会做决策**：知道下一步该干啥，干完了会检查结果对不对，不对还能自己纠错
- **会用工具**：能调 API、查数据库、甚至指挥别的 Agent 干活

### 1.2 怎么区分 Agent 和普通 AI 应用

很简单，问自己一个问题：**它只做了一件事，还是自己串了一串事？**

| 类型 | 特点 | 举个例子 |
|------|------|---------|
| 普通 AI 应用 | 你给它输入，它给你输出，完事 | 翻译、总结、情感分析 |
| Agent | 自己规划步骤，一步步干完，中间还会判断和纠错 | 自动处理客服工单、自动审批贷款 |

你让 AI 翻译一段话，那不叫 Agent。你让 AI 自己去查订单、判断问题、调退款接口、发邮件通知用户，这才叫 Agent。

### 1.3 Agent 的工作原理

说白了就是一个 **while 循环**：

```
while 事情没干完:
    1. 看看现在手上有什么信息
    2. 想想还缺什么，需要干什么
    3. 挑一个工具用一下
    4. 看看结果怎么样
    5. 决定：继续干 / 哪里不对改一下 / 搞定了收工 / 搞不定交给人
```

不管单 Agent 还是多 Agent，底层都是这个逻辑。

## 二、什么时候该用 Agent

### 2.1 三种适合用 Agent 的场景

OpenAI 的建议是：优先找那些**以前一直没法自动化的活**。

**1. 需要"脑子"的事**

比如判断一笔支付是不是欺诈。传统规则引擎就是对着清单逐条核对，Agent 更像一个老练的调查员，会综合各种信息做判断。

**2. 规则多到写不完的事**

比如贷款审批，不同地区、不同客户、不同贷款类型，规则全不一样。用 if-else 写到最后谁也维护不动。

**3. 输入乱七八糟的事**

比如从医生的手写处方里提取信息，字潦草、还有缩写，传统程序根本处理不了。

### 2.2 什么时候不该用

- **简单重复的活**：发模板邮件、批量录数据 → 用传统自动化工具就行
- **流程固定的活**：重置密码 → 写死规则就够了

**判断标准就一句话：这事需不需要"随机应变"？** 不需要的话，用 Agent 就是杀鸡用牛刀。

## 三、Agent 的三大核心组件

### 3.1 模型 —— Agent 的大脑

**选模型的思路：先用最好的，再慢慢降级。**

1. 先拿最强的模型跑出一个效果基准
2. 试试用便宜的小模型能不能达到差不多的效果
3. 不同的子任务可以用不同级别的模型

比如做一个健身 App：运动识别、简单问答这种，用小模型就行；个性化训练方案、复杂的饮食推理，再用强模型。**该省省，该花花。**

### 3.2 工具 —— Agent 的手和脚

没有工具的 Agent 就是个光会说的嘴炮。工具分三类：

| 类型 | 干什么的 | 举例 |
|------|---------|------|
| **数据工具** | 查信息 | 查数据库、读文件、搜网页 |
| **操作工具** | 干实事 | 发邮件、改记录、处理退款 |
| **编排工具** | 指挥别的 Agent | 让翻译 Agent 翻译、让审核 Agent 审核 |

**工具怎么写才好用：**

- 接口定义要清楚，别模棱两可
- 文档写详细点，AI 也是看文档干活的
- 多测试，工具出 bug 比 AI 胡说八道更致命

### 3.3 指令 —— Agent 的操作手册

**这是最重要的部分。** 指令写得模糊，Agent 就会瞎搞。

**怎么写好指令：**

**第一，从现有文档里提炼。** 公司的客服手册、操作流程、FAQ，这些现成的东西直接拿来改。

**第二，每一步都要写清楚。**

错误写法：`"帮客户处理问题"`

正确写法：
```
1. 问客户订单号
2. 调 get_order_status 接口查订单状态
3. 如果订单延迟了，给客户一个 9 折码
4. 如果客户没有订单号，让他提供邮箱或手机号来查
```

**第三，把意外情况想好。** 用户信息不全怎么办？问了不相关的问题怎么办？接口报错怎么办？都要提前写好。

**第四，用人话写。** 就像在教一个聪明的新同事，别整一堆术语。

## 四、编排方式 —— 一个人干还是团队干

### 4.1 核心原则：先用一个 Agent 干到极限

**不要一上来就搞多 Agent。** 多 Agent 看起来高大上，但复杂度会爆炸。先把一个 Agent 调好，实在不够用了再拆。

### 4.2 单 Agent 怎么玩

一个 Agent + 不断往上加工具，大部分场景够用了。

**一个技巧：用 Prompt 模板。** 不同场景不用写不同的 prompt，用一个模板 + 变量注入搞定：

```python
SYSTEM_PROMPT = """
你是一个{role}助手。
当前规则：{policy}
可用工具：{available_tools}
"""
```

新场景来了，改变量就行，不用重写整个流程。

### 4.3 什么时候该拆成多 Agent

- 工具太多了，尤其是功能相似容易搞混的工具
- 分支逻辑太复杂，一个 prompt 塞不下了
- 某个环节拖后腿，需要单独优化

**两种多 Agent 模式：**

**Manager 模式（老板带小弟）：** 一个总调度 Agent，下面挂几个专业 Agent。总调度决定把活派给谁，结果汇总后返回。

**Handoff 模式（击鼓传花）：** Agent 之间互相传递任务，A 干完传给 B，B 干完传给 C，每个 Agent 只管自己那一段。

### 4.4 对应到 LangGraph

| OpenAI 的概念 | LangGraph 怎么实现 |
|------------|---------------|
| 工作循环 | StateGraph 的图执行循环 |
| 工具调用 + 判断是否结束 | 条件边（conditional edges） |
| Manager 模式 | 中心节点调用子图 |
| Handoff 模式 | 节点之间路由跳转 |
| Prompt 模板 | State 里动态拼 system message |

## 五、安全护栏 —— 别让 Agent 搞破坏

### 5.1 核心思想：多层防御

一道防线不够，得多加几道。

### 5.2 三层护栏

**输入层（进来之前先过安检）：**
- 黑名单、输入长度限制、正则过滤
- AI 安全分类器：检测越狱攻击、prompt 注入
- 过滤掉和业务无关的请求
- 拦截敏感个人信息

**工具使用层（干活的时候盯着）：**
- 给工具分风险等级：查数据是低风险，删数据是高风险
- 高风险操作必须人工审批
- 加上频率限制和操作日志

**输出层（出去之前再检查一遍）：**
- 检查内容是否合规
- 过滤不安全的输出
- 用 JSON schema 约束输出格式

### 5.3 护栏不要阻塞主流程

一个好的做法：**护栏和 Agent 并行跑。** Agent 正常干活，护栏在旁边盯着，发现问题了再拦截。不要让 Agent 每一步都等护栏检查完才往下走，那样太慢了。

### 5.4 设上限，搞不定就交给人

给 Agent 设一个重试次数上限。比如连续 3 次都没理解用户的意思，别让它继续猜了，直接转人工。防止 Agent 陷入死循环越搞越离谱。

## 六、人工介入 —— 早期必须有人兜底

### 6.1 为什么一开始必须有人盯着

不是因为不信任 Agent，而是：

- Agent 在真实环境会遇到你没想到的情况，需要人发现问题
- 发现的问题可以反馈回去，让 Agent 越来越好
- 用户和团队需要时间建立对 Agent 的信心

### 6.2 两种介入方式

**1. 搞不定就上报：** Agent 处理不了的问题，交给人。

**2. 重要操作要审批：** 大额退款、删数据这种，Agent 干之前先问一下人。

### 6.3 逐步放手

```
阶段1：所有操作都要人工审批
阶段2：低风险的让 Agent 自己干，中高风险要审批
阶段3：只有高风险才要审批
阶段4：全自动，出问题了再自动上报
```

## 七、怎么一步步做起来

### 7.1 OpenAI 推荐的步骤

```
第1步：挑一个合适的场景
  ↓
第2步：用最强模型跑出一个效果基准
  ↓
第3步：搭一个单 Agent + 几个核心工具
  ↓
第4步：找真实用户测试，收集反馈
  ↓
第5步：加上护栏和兜底机制
  ↓
第6步：真的需要的时候再拆成多 Agent
  ↓
第7步：优化成本（换便宜模型）和速度
```

### 7.2 评估（Eval）是一切的基础

指南反复强调这一点：**没有评估体系，你改了 prompt、换了模型、加了工具，根本不知道是变好了还是变差了。** 所有优化都得有数据支撑，否则就是瞎调。

## 九、总结

### 一句话版

**Agent = 能干活的模型 + 好用的工具 + 清晰的指令 + 合理的编排 + 安全护栏 + 人类兜底**

### 十个核心要点

1. **Agent 不是聊天机器人**：它是自己能干活的系统，不只是回答问题
2. **别滥用**：只在需要"随机应变"的场景用 Agent，简单的事用传统方案
3. **三大基石**：模型 + 工具 + 指令，缺一不可
4. **指令最重要**：写不清楚指令，Agent 就是定时炸弹
5. **先单后多**：一个 Agent 能搞定就别拆成多个
6. **模板化管理**：用 prompt 模板 + 变量注入，别每个场景写一套
7. **工具贵精不贵多**：几个定义清晰的工具，比一堆模糊的工具好使
8. **护栏必须有**：输入、工具使用、输出三层防护
9. **早期必须有人盯着**：人工介入不是不信任，是学习机制
10. **评估驱动优化**：没有 eval，所有优化都是瞎蒙
